# API-005 — SQLite Adapter and Migrations

**Implementer inference:** Low
**Prerequisites:** `FEAS-004`, `API-001`, and `SEAM-001` approved.

## Frozen inputs

Use ADR-0012, the exact SQLite source/provenance frozen by `FEAS-004`, the
schema documentation rules in `ARCHITECTURE.md`, and only the API adapter seams
authorized by `PLAN-002`. Do not reintroduce PostgreSQL or pg.zig.

## Objective

Implement the SQLite connection boundary, safe migrations, synchronized
implemented-schema DBML, and real health probing.

## Procedure

1. Add SQLite native build wiring using only the approved official source and
   compile options from `FEAS-004`.
2. Open only root-contained database paths and enable WAL, foreign keys, and the
   frozen busy timeout.
3. Add a migration table with version, name, SHA-256, and applied timestamp.
4. Apply migrations transactionally in strict order.
5. Reject missing versions, checksum drift, and a database newer than the
   executable.
6. Expose sanitized health/probe behavior that maps only to the frozen SQLite
   public error codes.
7. Create or update `docs/database/sqlite.dbml` and `docs/database/README.md`
   in this same revision for every migration change. Include every table,
   column, type affinity, constraint, index, trigger, and relation; SQL remains
   authoritative.
8. Validate DBML with a coordinator-approved compatibility check. Do not add an
   unpinned DBML package or CLI.
9. Add focused unit and self-test coverage for fresh, retained, checksum-drift,
   newer-schema, busy, invalid-root, probe, and close behavior.

## Acceptance evidence

- SQLite adapter and migration tests through the aggregate build.
- Runtime self-test evidence from repository root and `services/api`.
- Schema-change inventory matched to the final migration revision.
- Successful DBML compatibility-check output and a table/column/type/
  constraint/index/trigger/relation comparison against final applied
  migrations.
- Sensitive-content scan and `git diff --check`.

## Reviewer focus

Inspect transaction boundaries, WAL/busy-timeout setup, path containment,
resource closure, migration corruption cases, and reject any migration revision
whose implemented DBML is absent or stale.
