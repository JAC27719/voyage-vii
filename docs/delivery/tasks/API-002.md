# API-002 — PostgreSQL Adapter and Migrations

**Implementer inference:** Low  
**Prerequisites:** `API-001` approved.

## Frozen inputs

Use `DEPENDENCY-PINS.md`, `CONTRACTS.md`, and `TIMEOUTS.md`. Use only the API-001 static seam. Stop if `build.zig`, `main.zig`, aggregate test registration, or another task's path must change.

## Objective

Implement the pg.zig connection boundary, safe migrations, synchronized implemented-schema DBML, and real health probing.

## Procedure

1. Build a pool with maximum size two and the exact connect, acquisition, and query deadlines in `TIMEOUTS.md`.
2. Add a migration table with version, name, SHA-256, and applied timestamp.
3. Acquire a PostgreSQL advisory lock before migration.
4. Apply each migration transactionally in strict order.
5. Reject missing versions, checksum drift, and a database newer than the executable.
6. Implement a three-second `SELECT 1` probe and explicit connection cleanup.
7. Log operation name and timing only—never SQL parameter values or credentials.
8. Create or update `docs/database/postgresql.dbml` and `docs/database/README.md` in this same revision for every migration change. Include every table, column, type, constraint, index, and relation; SQL remains authoritative.
9. Validate DBML with the coordinator-approved compatible parser and test migration-to-DBML consistency. No later documentation task may repair schema drift separately.

## Acceptance evidence

- Fresh, retained, concurrent, rollback, checksum-drift, version-gap, and newer-schema tests.
- Pool exhaustion and unavailable-server tests.
- Log scan for SQL values and credentials.
- Schema-change inventory matched to the final migration revision.
- Successful DBML parser output and a table/column/constraint/index/relation comparison against the final applied migrations.
- `zig build test` succeeds through API-001's static aggregate registration; `git diff --check` passes.

## Reviewer focus

Inspect transaction and lock boundaries, verify all error branches release resources, reproduce migration corruption cases, and reject any migration revision whose implemented DBML is absent or stale.
